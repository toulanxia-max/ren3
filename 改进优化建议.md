## 🚨 高优先级问题（必须立即解决）

### 1.1 安全风险 - 硬编码凭证

**问题描述**:
- `backend/.env`中数据库密码为简单密码`111111`
- JWT密钥使用默认值`your_jwt_secret_key_here_change_in_production`
- `docker-compose.yml`中MySQL root密码为`rootpassword`
- 所有配置在生产环境均未修改

**风险等级**: 🔴 **高风险**

**影响**:
- 数据库可能被未授权访问
- JWT令牌可能被伪造
- 系统存在严重安全漏洞

**解决方案**:
```bash
# 1. 生成强密码和密钥
openssl rand -base64 32  # 生成32位随机密钥

# 2. 创建生产环境配置文件
cp backend/.env backend/.env.production
# 修改以下配置：
# DB_PASSWORD=强密码（至少16位，包含大小写字母、数字、特殊字符）
# JWT_SECRET=生成的随机密钥
# NODE_ENV=production

# 3. 更新docker-compose.yml
# 修改mysql服务的MYSQL_ROOT_PASSWORD为强密码
```

### 1.2 前端安全 - 令牌存储方式

**问题描述**:
- 使用`localStorage`存储JWT令牌
- 易受XSS攻击，令牌可能被恶意脚本窃取
- 缺乏令牌刷新机制

**风险等级**: 🟡 **中风险**

**解决方案**:
```javascript
// 方案1：使用httpOnly cookies（推荐）
// 后端设置cookie
res.cookie('token', token, {
  httpOnly: true,
  secure: process.env.NODE_ENV === 'production',
  sameSite: 'strict',
  maxAge: 7 * 24 * 60 * 60 * 1000 // 7天
});

// 方案2：实现令牌刷新机制
// 添加刷新令牌端点，短期访问令牌+长期刷新令牌
```

### 1.3 数据库权限配置

**问题描述**:
- 数据库初始化脚本创建用户`'ninja_clan_user'@'%'`，允许任意IP访问
- 生产环境未限制数据库访问来源

**风险等级**: 🟡 **中风险**

**解决方案**:
```sql
-- 限制数据库用户访问来源
CREATE USER 'ninja_clan_user'@'应用服务器IP' IDENTIFIED BY '强密码';
-- 或使用Docker内部网络
CREATE USER 'ninja_clan_user'@'172.%.%.%' IDENTIFIED BY '强密码';
```

```env
FRONTEND_URL=http://localhost:3000

# .env.production (生产环境)
PORT=3001
API_BASE_URL=/api
FRONTEND_URL=https://your-domain.com
```

### 2.2 数据库同步策略

**问题描述**:
- 开发环境使用`sequelize.sync({ alter: true })`
- 可能造成数据丢失或结构冲突
- 缺乏版本控制的数据库迁移

**解决方案**:
```bash
# 1. 创建数据库迁移系统
npm install --save-dev sequelize-cli

# 2. 创建迁移文件
npx sequelize-cli migration:generate --name create-users-table

# 3. 开发环境使用迁移，禁用自动同步
if (config.app.nodeEnv === 'development') {
  // await sequelize.sync({ alter: true }); // 移除
  // 运行迁移
  await sequelize.runMigrations();
}
```

### 2.3 日志管理混乱

**问题描述**:
- 根目录存在多个日志文件（`backend_*.log`, `frontend_*.log`）
- 缺乏统一的日志轮转和归档策略
- 日志格式不统一

**解决方案**:
```javascript
// 配置统一的Winston日志系统
const logger = winston.createLogger({
  level: process.env.LOG_LEVEL || 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  transports: [
    new winston.transports.File({
      filename: 'logs/error.log',
      level: 'error',
      maxsize: 5242880, // 5MB
      maxFiles: 5
    }),
    new winston.transports.File({
      filename: 'logs/combined.log',
      maxsize: 5242880,
      maxFiles: 10
    })
  ]
});

// 生产环境添加日志轮转
if (process.env.NODE_ENV === 'production') {
  logger.add(new winston.transports.DailyRotateFile({
    filename: 'logs/application-%DATE%.log',
    datePattern: 'YYYY-MM-DD',
    maxSize: '20m',
    maxFiles: '30d'
  }));
}
```

### 2.4 API安全防护缺失

**问题描述**:
- 缺乏API请求频率限制
- 未实现输入验证和消毒
- 缺少SQL注入防护

**解决方案**:
```javascript
// 1. 添加速率限制
const rateLimit = require('express-rate-limit');
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15分钟
  max: 100, // 每个IP限制100个请求
  message: '请求过于频繁，请稍后再试'
});
app.use('/api/', limiter);

// 2. 添加Helmet安全头
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],
      styleSrc: ["'self'", "'unsafe-inline'"],
      imgSrc: ["'self'", "data:", "https:"]
    }
  }
}));

// 3. 输入验证和消毒
const { body, sanitizeBody } = require('express-validator');
app.post('/api/register', [
  body('username').trim().escape(),
  body('email').normalizeEmail(),
  // ...其他验证规则
]);
```
